## 1. 開発者特性別の予測成功率

本分析では、**最高性能モデル（train: 0-3m → eval: 6-9m, AUC-ROC 0.910）**を使用して、開発者の特性別に予測精度を詳細に分解する。

### 1.1 経験レベル別の予測精度

開発者を過去の依頼数で4つのレベルに分類し、各レベルでの予測精度を分析した。

#### 【初心者】依頼数 < 10回（16名、38.1%）

```
総数: 16名
正例（実際承諾）: 0名 (0.0%)
負例（実際拒否）: 16名 (100.0%)

予測結果:
  ✓ 正しく拒否を予測（TN）: 16名 / 16名 = 100.0%
  ✗ 誤って承諾と予測（FP）: 0名 / 16名 = 0.0%

性能指標:
  Accuracy: 1.000（完璧）
  Precision: 0.000（承諾予測なし）
  Recall: 0.000（承諾者なし）
```

**分析**:
- **完璧な負例予測**：初心者16名全員が実際に拒否し、モデルも全員を正しく「拒否する」と予測
- **承諾者ゼロ**：6-9m期間の初心者は誰も承諾しなかった
- **モデルの学習**：経験不足（依頼数 < 10）は、継続性の極めて強い負のシグナル
- **初期の3ヶ月間で10回未満の経験しか積めなかった開発者は、その後も参加しない**

**実務への示唆**:
- 初心者（依頼数 < 10）への推薦は避けるべき
- むしろ、彼らに**経験を積む機会を集中的に提供**し、10回以上の経験を目指すべき

---

#### 【中級者】依頼数 10-100回（13名、31.0%）

```
総数: 13名
正例（実際承諾）: 3名 (23.1%)
負例（実際拒否）: 10名 (76.9%)

予測結果:
  ✓ 正しく承諾を予測（TP）: 2名 / 3名 = 66.7%
  ✓ 正しく拒否を予測（TN）: 6名 / 10名 = 60.0%
  ✗ 誤って承諾と予測（FP）: 4名 / 10名 = 40.0%
  ✗ 誤って拒否と予測（FN）: 1名 / 3名 = 33.3%

性能指標:
  Accuracy: 0.615（やや低い）
  Precision: 0.333（推薦の3割しか的中しない）
  Recall: 0.667（承諾者の2/3は捕捉）
```

**分析**:
- **最も予測が難しいグループ**：Accuracy 0.615は4グループ中最低
- **False Positive率40%**：10名の負例のうち4名を誤って「承諾する」と予測
- **不安定な行動パターン**：中級者は、まだ行動パターンが確立していない「移行期」にある
- 一部は上級者へと成長し、一部は離脱する。この分岐点を初期3ヶ月のデータから予測するのは困難

**実務への示唆**:
- 中級者への推薦は**慎重に**行うべき
- 推薦する場合は、**他の指標（過去受諾率、最近の活動頻度など）を組み合わせる**
- むしろ、中級者を**上級者へと育成する施策**に注力すべき

---

#### 【上級者】依頼数 100-500回（9名、21.4%）

```
総数: 9名
正例（実際承諾）: 7名 (77.8%)
負例（実際拒否）: 2名 (22.2%)

予測結果:
  ✓ 正しく承諾を予測（TP）: 7名 / 7名 = 100.0%（完璧）
  ✓ 正しく拒否を予測（TN）: 0名 / 2名 = 0.0%
  ✗ 誤って承諾と予測（FP）: 2名 / 2名 = 100.0%

性能指標:
  Accuracy: 0.778（良好）
  Precision: 0.778（推薦の78%が的中）
  Recall: 1.000（承諾者を全員捕捉）
```

**分析**:
- **承諾者の完璧な捕捉**：実際に承諾した7名全員を正しく予測
- **高い正例率77.8%**：上級者の大多数が実際に承諾している
- **False Positive 2名**：負例2名を誤って「承諾する」と予測したが、サンプル数が少ないため大きな問題ではない
- **モデルの強み**：100-500回の経験を持つ開発者の行動パターンを高精度で捉えている

**実務への示唆**:
- 上級者は**最優先で推薦すべき**グループ
- Precision 0.778は実用的に十分高い
- 彼らへの推薦は、高い成功率が期待できる

---

#### 【エキスパート】依頼数 ≥ 500回（4名、9.5%）

```
総数: 4名
正例（実際承諾）: 3名 (75.0%)
負例（実際拒否）: 1名 (25.0%)

予測結果:
  ✓ 正しく承諾を予測（TP）: 3名 / 3名 = 100.0%（完璧）
  ✓ 正しく拒否を予測（TN）: 1名 / 1名 = 100.0%（完璧）
  ✗ 誤り: 0名

性能指標:
  Accuracy: 1.000（完璧）
  Precision: 1.000（完璧）
  Recall: 1.000（完璧）
```

**分析**:
- **完璧な予測**：4名全員を正しく予測（承諾3名、拒否1名）
- **極めて安定した行動パターン**：500回以上の経験を持つエキスパートは、行動が高度に予測可能
- **コアコントリビューター**：このレベルの開発者は、プロジェクトの中核を担っている
- サンプル数は少ないが、モデルの予測能力の高さを示す

**実務への示唆**:
- エキスパートは**絶対に推薦すべき**
- 彼らの時間は貴重なので、**本当に重要なレビュー**に絞って依頼すべき
- 彼らを失うことは、プロジェクトにとって大きな損失

### 1.2 過去受諾率別の予測精度

開発者を過去の受諾率で5つのレベルに分類し、各レベルでの予測精度を分析した。

#### 【未承諾】過去受諾率 0%（9名、21.4%）

```
総数: 9名
正例（実際承諾）: 0名 (0.0%)
負例（実際拒否）: 9名 (100.0%)

予測結果:
  ✓ 正しく拒否を予測（TN）: 8名 / 9名 = 88.9%
  ✗ 誤って承諾と予測（FP）: 1名 / 9名 = 11.1%

性能指標:
  Accuracy: 0.889（高い）
```

**分析**:
- **過去に一度も承諾していない開発者は、将来も承諾しない**
- モデルは9名中8名を正しく「拒否する」と予測
- False Positive 1名は、おそらく他の特徴量（活動頻度など）が高かった例外的ケース

**実務への示唆**:
- 過去受諾率0%の開発者への推薦は**避けるべき**
- 彼らは、このプロジェクトに関心がない、または時間がないと考えられる

---

#### 【低受諾】過去受諾率 1-10%（10名、23.8%）

```
総数: 10名（平均過去受諾率: 5.4%）
正例（実際承諾）: 2名 (20.0%)
負例（実際拒否）: 8名 (80.0%)

予測結果:
  ✓ 正しく承諾を予測（TP）: 2名 / 2名 = 100.0%（完璧）
  ✓ 正しく拒否を予測（TN）: 4名 / 8名 = 50.0%
  ✗ 誤って承諾と予測（FP）: 4名 / 8名 = 50.0%

性能指標:
  Accuracy: 0.600
  Precision: 0.333
  Recall: 1.000
```

**分析**:
- **Recall 1.000**：実際に承諾した2名を全員捕捉（見逃しゼロ）
- **Precision 0.333**：推薦した6名のうち、実際に承諾したのは2名のみ
- **False Positive率50%**：負例8名のうち4名を誤って推薦
- **低受諾率でも承諾する人がいる**：モデルは彼らを捉えているが、過剰推薦の傾向

**実務への示唆**:
- 低受諾率の開発者は**慎重に推薦**
- 推薦する場合は、**他の強いシグナル（経験レベル、最近の活動など）**がある場合のみ
- False Positive率50%は実用上問題

---

#### 【中受諾】過去受諾率 10-25%（12名、28.6%）

```
総数: 12名（平均過去受諾率: 18.9%）
正例（実際承諾）: 7名 (58.3%)
負例（実際拒否）: 5名 (41.7%)

予測結果:
  ✓ 正しく承諾を予測（TP）: 7名 / 7名 = 100.0%（完璧）
  ✓ 正しく拒否を予測（TN）: 5名 / 5名 = 100.0%（完璧）
  ✗ 誤り: 0名

性能指標:
  Accuracy: 1.000（完璧）
  Precision: 1.000（完璧）
  Recall: 1.000（完璧）
```

**分析**:
- **完璧な予測**：12名全員を正しく予測（承諾7名、拒否5名）
- **最も予測しやすいグループ**：過去受諾率10-25%は、モデルが最も得意とする範囲
- **「選択的だが確実」な開発者**：全ての依頼に応じるわけではないが、適切な依頼には確実に応じる
- このレベルの受諾率が、**最も安定した行動パターン**を示す

**実務への示唆**:
- **最も推薦すべきグループ**
- 過去受諾率10-25%の開発者は、高い確率で承諾する
- 彼らは「選択的なレビュアー」であり、**適切なタスクを選んで推薦する**ことで高い成功率を達成できる

---

#### 【高受諾】過去受諾率 25-40%（6名、14.3%）

```
総数: 6名（平均過去受諾率: 30.3%）
正例（実際承諾）: 4名 (66.7%)
負例（実際拒否）: 2名 (33.3%)

予測結果:
  ✓ 正しく承諾を予測（TP）: 3名 / 4名 = 75.0%
  ✓ 正しく拒否を予測（TN）: 1名 / 2名 = 50.0%
  ✗ 誤って承諾と予測（FP）: 1名 / 2名 = 50.0%
  ✗ 誤って拒否と予測（FN）: 1名 / 4名 = 25.0%

性能指標:
  Accuracy: 0.667
  Precision: 0.750
  Recall: 0.750
```

**分析**:
- **良好だが完璧ではない**：Accuracy 0.667、Precision/Recall 0.750
- **False Negative 1名**：高受諾率（30.3%）にも関わらず、承諾すると予測されなかった例
- おそらく、**最近の活動頻度が低下**していた可能性
- モデルは「過去の高受諾率」よりも「最近の行動パターン」を重視する傾向

**実務への示唆**:
- 高受諾率の開発者も**推薦対象**だが、完璧ではない
- **最近の活動状況を確認**してから推薦すべき
- 過去の受諾率だけでなく、最近の行動トレンドが重要

---

#### 【超高受諾】過去受諾率 > 40%（5名、11.9%）

```
総数: 5名（平均過去受諾率: 57.6%）
正例（実際承諾）: 0名 (0.0%)
負例（実際拒否）: 5名 (100.0%)

予測結果:
  ✓ 正しく拒否を予測（TN）: 5名 / 5名 = 100.0%（完璧）
  ✗ 誤り: 0名

性能指標:
  Accuracy: 1.000（完璧）
```

**分析**:
- **極めて興味深い発見**：過去受諾率が**57.6%と極めて高い**にも関わらず、5名全員が6-9m期間に拒否
- モデルは全員を正しく「拒否する」と予測
- **過去の高受諾率は、将来の承諾を保証しない**
- おそらく、これらの開発者は：
  - 活動間隔が長期化している（**離脱の兆候**）
  - 最近の活動頻度が低下している
  - プロジェクトへの関心が薄れている

**実務への示唆**:
- **過去の受諾率だけでは判断できない**
- 超高受諾率（> 40%）でも、**最近の行動トレンド**が重要
- モデルは、「過去の栄光」よりも「最近の状態」を重視している

### 1.3 2次元クロス分析：経験 × 受諾率

開発者を経験レベルと過去受諾率の2軸で分類し、最も予測精度が高い組み合わせを特定した。

#### 【完璧な予測（Accuracy 1.000）のセグメント】

**上級者 × 中受諾率（5名）**:
```
総数: 5名
実際承諾: 5名（100%）
正しく予測: 5名（100%）

特徴:
  - 依頼数: 100-500回（豊富な経験）
  - 過去受諾率: 10-25%（選択的だが確実）
  - 予測成功率: 100%
```

**エキスパート × 中受諾率（1名）**:
```
総数: 1名
実際承諾: 1名（100%）
正しく予測: 1名（100%）
```

**エキスパート × 高受諾率（2名）**:
```
総数: 2名
実際承諾: 2名（100%）
正しく予測: 2名（100%）
```

**分析**:
- **「経験豊富 × 選択的な受諾率」が最強の組み合わせ**
- 100回以上の経験 × 10-25%の受諾率 = **最も予測しやすく、実際に承諾する確率も高い**
- この組み合わせは、「プロジェクトにコミットしているが、時間を選択的に使う」という理想的なパターン

## 2. 期間別の特徴量重要度変化と予測精度の関係

### 2.1 特徴量重要度の時間的推移

#### 【総レビュー数】経験の価値は時とともに減衰する

```
0-3m期間: +0.0316 ████████████████████████████████ 最高
3-6m期間: +0.0176 ████████████████
6-9m期間: +0.0102 ████████
9-12m期間: +0.0066 ████
```

**変化率**: 初期→長期で **79%減少**（0.0316 → 0.0066）

**考察**:
- **初期（0-3m）**: 経験が**圧倒的に重要**
  - 新規参加者にとって、「何回レビューしたか」が全てを決定する
  - 10回 vs 20回の差は、継続性に決定的な影響
  - この期間の総レビュー数は、長期的な継続性を予測する**最強の単一指標**

- **長期（9-12m）**: 経験の重要度は**1/5に減衰**
  - 長期参加者にとって、「500回経験したか100回か」はあまり重要ではない
  - **最近の行動パターン**の方が重要になる
  - 「過去の栄光」だけでは継続性を保証できない

**なぜこの変化が起きるのか**:
1. **初期**: 経験は**スキル、自信、プロジェクト理解**の全ての基盤
   - 経験不足 → レビューが怖い → 離脱
   - 経験豊富 → レビューに自信 → 継続

2. **長期**: 経験は**既に十分に蓄積**されている
   - 100回でも500回でも、基本的なスキルは習得済み
   - 継続性を決めるのは、経験量ではなく**モチベーション、社会的つながり、最近の活動状況**

**予測精度への影響**:
- **0-3m訓練モデル**: 経験レベルで開発者を高精度に分類
  - エキスパート（≥500回）: Accuracy 1.000
  - 初心者（<10回）: Accuracy 1.000（全員拒否と予測）
  - 経験量が決定的な分類基準として機能

- **9-12m訓練モデル**: 経験レベルだけでは予測困難
  - 経験豊富でも離脱する人が増える
  - 最近の活動パターン、協力度が重要に

---

#### 【協力度】時間とともに重要性が増す社会的つながり

```
0-3m期間: +0.0156 ████████████████
3-6m期間: +0.0091 ████████
6-9m期間: +0.0131 █████████████
9-12m期間: +0.0146 ██████████████ 最高
```

**変化率**: 初期→長期で **6%減少** → **61%増加**（U字型）

**考察**:
- **初期（0-3m）**: 協力度は**重要だが最優先ではない**
  - 新規参加者にとって、まず「経験を積む」ことが最優先
  - 協力度+0.0156 vs 総レビュー数+0.0316（約半分）
  - しかし、初期の協力的な姿勢は継続性を促進

- **中期（3-6m）**: 協力度の重要度が**一時的に低下**
  - 「独り立ち」の時期
  - 個人の自律的な活動が重要になる
  - 経験、活動頻度など、個人指標が相対的に重視される

- **長期（9-12m）**: 協力度が**最高レベルに到達**
  - +0.0146（全期間で最高）
  - 長期継続には**社会的つながりが不可欠**
  - 技術的なスキルよりも「コミュニティへの帰属感」が重要に

**なぜこの変化が起きるのか**:

1. **初期**: 「仲間を見つける」フェーズ
   - 協力的なレビューを通じて、プロジェクトに統合される
   - しかし、まだ「経験を積む」ことが最優先

2. **中期**: 「独り立ち」フェーズ
   - 既にプロセスを理解し、自分なりのスタイルを確立
   - 他者との協力も大切だが、それ以上に**自分の判断で独立して行動できる能力**が重要

3. **長期**: 「コミュニティへの帰属」フェーズ
   - 1年近く参加を続けている開発者にとって、プロジェクトは単なる「タスクをこなす場所」ではなく**コミュニティ**
   - 他の開発者との関係性、共有された目標、一緒に問題を解決してきた経験
   - これらの社会的な要素が、継続的な参加を支えている

**予測精度への影響**:
- **0-3m訓練モデル**: 協力度の影響は中程度
  - 協力度が高い開発者は継続しやすいが、経験量の方が強いシグナル

- **9-12m訓練モデル**: 協力度が**最重要指標**
  - 長期参加者の中で、協力度の高い人は継続し、低い人は離脱
  - 経験量では区別できない（全員が経験豊富）ため、協力度が決定的な分類基準に

**実測データでの確認**:
- 9-12m訓練モデルの最高性能評価期間: 3-6m（AUC-ROC 0.715）
- しかし、この性能は0-3m訓練モデル（AUC-ROC 0.910）より大幅に低い
- **理由**: 9-12m期間は経験の重要度が低く、協力度が高い
  - しかし、協力度は「行動特徴量」であり、LSTMで学習される時系列パターン
  - 評価時には**集約特徴量**を使うため、時系列の協力度パターンを完全には捉えきれない
  - 結果として、9-12m訓練モデルの予測精度は相対的に低下

2. **中期の緩和**: 安定期の柔軟性
   - 既に参加実績があり、プロジェクトへのコミットメントを示している
   - 一時的な不在は「休憩」として許容される
   - 「戻ってくる」という信頼感

3. **長期の再監視**: 変化のサイン
   - 1年近く参加してきた開発者が、突然活動間隔を長期化
   - これは単なる「休憩」ではなく、**状況の変化**を示唆
   - 離脱の前兆として注意が必要

**予測精度への影響**:

  - 中期の「許容される休憩」を学習
  - しかし、これが逆に予測精度に影響する可能性
  - 「一時的な休憩」と「離脱の兆候」を区別できない

  - 「500回経験した」という事実は変わらない
  - 長期にわたって安定した予測因子として機能

  - 長期的な予測には不向き

  - 時系列の詳細なパターンを評価時に完全には再現できない
  - 結果として、協力度に依存するモデルは評価時の精度が低下

  - 単純な分類が困難
  - 「経験豊富だが協力度が低い」「経験は中程度だが協力度が高い」など、複雑な組み合わせ

**仮説1: 最近の活動頻度の低下**
- モデルは「最近の活動頻度」を重視（0-3m期間の重要度 +0.0174）
- おそらく、0-3m期間のデータを見ると、**活動頻度が低下していた**
- モデルは「最近活動が減っている → 離脱の兆候 → 拒否する」と判断

**仮説2: 平均活動間隔の長期化**
- 平均活動間隔の重要度: -0.0165（強い負）
- おそらく、依頼の間隔が長くなっていた
- モデルは「間隔が長い → 継続性が低い → 拒否する」と判断

**仮説3: 協力度の低さ**
- 協力度の重要度: +0.0156
- おそらく、単独で活動することが多く、協力的なレビューが少なかった
- モデルは「協力度が低い → 社会的つながりが弱い → 離脱しやすい」と判断

**実際には承諾した理由**:
- 一時的な不活発期間を経て、**活動を再開した**
- モデルにとって、このような「復活パターン」は予測困難
- 過去の高い受諾率（34.8%）が、最終的には継続意欲を示していた

**この見逃しから学ぶべきこと**:
- **過去の高受諾率は、一時的な不活発期間があっても、重要なシグナル**
- モデルは「最近の行動」を重視しすぎる傾向がある
- 高受諾率（> 30%）の開発者は、一時的に活動が減っても、**推薦対象として考慮すべき**

### 3.3 True Positive（正しく承諾を予測）: 12件

#### 全体的な傾向

```
総数: 12件（正例13件中、Recall 92.3%）
平均過去受諾率: 19.8%
平均依頼数: 346.7回（高い）
平均予測確率: 0.4723（高い）
```

**分析**:
- **Recall 92.3%**: 実際に承諾した13名のうち12名を正しく予測（見逃しは1名のみ）
- **平均過去受諾率19.8%**: 「中受諾」カテゴリ（10-25%）に該当
  - このカテゴリの予測精度はAccuracy 1.000（完璧）
- **平均依頼数346.7回**: 「上級者」レベル（100-500回）
  - このカテゴリの予測精度はAccuracy 0.778、Recall 1.000

**成功の要因**:
1. **経験豊富 × 中受諾率の組み合わせ**
   - 上級者12名中、おそらく大部分が中受諾率
   - この組み合わせは、モデルが最も得意とするパターン

2. **高い予測確率（0.4723）**
   - False Positive（0.4636）よりも高い
   - False Negative（0.4513）よりも高い
   - モデルは、真の承諾者を**明確に高く評価**している

3. **安定した行動パターン**
   - 経験豊富で、選択的だが確実に応じる
   - 活動頻度が安定している
   - 協力度も一定レベルを維持

### 3.4 予測失敗の5つのメカニズム

#### メカニズム1: 経験の過大評価

**発生条件**: 依頼数は多いが、受諾率が極めて低い開発者

**例**: takanattie@gmail.com（451回依頼、受諾率5.3%）

**なぜ起きるのか**:
- 0-3m訓練モデルは、総レビュー数の重要度が+0.0316（最高）
- 経験量を最重要視するため、受諾率の低さを過小評価

**対策**:
- **受諾率と経験量の相互作用**を学習させる
- 「高経験 × 低受諾率」は特別なパターンとして認識すべき
- または、推薦時に**受諾率 < 10%はフィルタリング**

---

#### メカニズム2: 中級者の不安定性

**発生条件**: 依頼数10-100回の「移行期」にある開発者

**例**: wangzihao@yovole.com（44回依頼、受諾率9.1%）

**なぜ起きるのか**:
- 中級者は「成長中」と「離脱中」が混在
- 初期3ヶ月のデータだけでは、この分岐を予測困難

**対策**:
- 中級者への推薦は**慎重に**
- 他の強いシグナル（中受諾率、高い協力度など）がある場合のみ推薦
- または、中級者を**育成対象**として、経験を積む機会を優先的に提供

---

#### メカニズム3: 最近の活動トレンドの見逃し

**発生条件**: 過去は活発だったが、最近活動が減少している開発者

**例**: False Positive 6件全体の傾向

**なぜ起きるのか**:
- 0-3m訓練モデルは、0-3m期間のデータで訓練
- 6-9m期間を評価する際、その間の変化（3-6m期間の活動減少など）を捉えられない

**対策**:
- **3-6m訓練モデルを使用**する方が、6-9m期間の予測に適している可能性
  - 実際、3-6m → 6-9mはAUC-ROC 0.894（0-3m → 6-9mの0.910に近い）
- または、**リアルタイム推薦**では、最新3-6ヶ月のデータで訓練したモデルを使用

---

#### メカニズム4: 協力度の時系列パターンの限界

**発生条件**: 長期期間（9-12m訓練）で協力度が重要になるが、評価時に捉えきれない

**例**: 9-12m訓練モデルの低性能（AUC-ROC 0.693-0.715）

**なぜ起きるのか**:
- 協力度は**時系列パターン**（LSTM訓練）
- しかし、評価時には**集約特徴量**（平均協力度など）を使用
- 時系列の詳細なパターン（「最近協力度が上がった」など）を再現できない

**対策**:
- **評価時もLSTMを使用**する（現在は集約特徴量のみ）
- または、協力度の**時系列的な変化**を明示的に特徴量化
  - 「3ヶ月前の協力度」「1ヶ月前の協力度」の差分など

---

#### メカニズム5: 「復活パターン」の予測困難性

**発生条件**: 一時的に活動が減少したが、再び活動を再開する開発者

**例**: kchamart@redhat.com（False Negative、唯一の見逃し）

**なぜ起きるのか**:
- モデルは「最近の活動減少 = 離脱の兆候」と学習
- しかし、実際には「一時的な休息」の場合もある
- 過去の高い受諾率（34.8%）は、長期的な継続意欲を示していた

**対策**:
- **過去の受諾率 > 30%の開発者は、特別扱い**
- たとえ最近の活動が減少していても、推薦候補として残す
- または、「休息後の復活パターン」を明示的に学習
  - 「過去の高受諾率 × 最近の低活動頻度」の組み合わせを、特別なパターンとして認識

### 4.1 推薦アルゴリズムの設計

#### ステップ1: 開発者をセグメント化

```python
def segment_developer(developer):
    """開発者を7つのセグメントに分類"""

    request_count = developer['history_request_count']
    acceptance_rate = developer['history_acceptance_rate']

    # 経験レベル
    if request_count < 10:
        exp_level = 'beginner'
    elif request_count < 100:
        exp_level = 'intermediate'
    elif request_count < 500:
        exp_level = 'advanced'
    else:
        exp_level = 'expert'

    # 受諾率レベル
    if acceptance_rate == 0:
        acc_level = 'none'
    elif acceptance_rate < 0.10:
        acc_level = 'low'
    elif acceptance_rate < 0.25:
        acc_level = 'medium'
    elif acceptance_rate < 0.40:
        acc_level = 'high'
    else:
        acc_level = 'very_high'

    return {
        'exp_level': exp_level,
        'acc_level': acc_level,
        'segment': f"{exp_level}_{acc_level}"
    }
```

#### ステップ2: セグメント別の推薦戦略

```python
# 推薦優先度の定義
RECOMMENDATION_STRATEGY = {
    # 【強く推薦】Success Rate > 90%
    'expert_medium': {'priority': 1, 'expected_success': 1.000},
    'expert_high': {'priority': 1, 'expected_success': 1.000},
    'advanced_medium': {'priority': 1, 'expected_success': 1.000},

    # 【推薦】Success Rate 75-90%
    'advanced_any': {'priority': 2, 'expected_success': 0.778},
    'expert_low': {'priority': 2, 'expected_success': 0.800},
    'intermediate_medium': {'priority': 2, 'expected_success': 0.750},

    # 【条件付き推薦】Success Rate 60-75%
    'intermediate_low': {'priority': 3, 'expected_success': 0.667},
    'advanced_high': {'priority': 3, 'expected_success': 0.750},

    # 【推薦しない】Success Rate < 60% or 承諾者ゼロ
    'beginner_any': {'priority': 999, 'expected_success': 0.000},
    'any_none': {'priority': 999, 'expected_success': 0.000},
    'any_very_high': {'priority': 999, 'expected_success': 0.000},
    'intermediate_high': {'priority': 999, 'expected_success': 0.000},
}
```

#### ステップ3: モデル予測との統合

```python
def recommend_reviewers(pr, candidates, model, top_n=5):
    """レビュアーを推薦する統合アルゴリズム"""

    recommendations = []

    for candidate in candidates:
        # セグメント化
        segment_info = segment_developer(candidate)
        segment = segment_info['segment']

        # 基本フィルタリング
        if segment_info['acc_level'] == 'none':
            continue  # 過去受諾率0%は除外
        if segment_info['exp_level'] == 'beginner':
            continue  # 初心者は除外
        if segment_info['acc_level'] == 'very_high':
            continue  # 超高受諾率（離脱の兆候）は除外

        # モデル予測
        pred_prob = model.predict(candidate)

        # セグメント別の期待成功率
        strategy = RECOMMENDATION_STRATEGY.get(segment, {'priority': 99, 'expected_success': 0.5})

        # 統合スコア（モデル予測 × セグメント期待値）
        final_score = pred_prob * strategy['expected_success']

        recommendations.append({
            'email': candidate['email'],
            'segment': segment,
            'model_prob': pred_prob,
            'expected_success': strategy['expected_success'],
            'final_score': final_score,
            'priority': strategy['priority']
        })

    # 優先度とスコアでソート
    recommendations.sort(key=lambda x: (x['priority'], -x['final_score']))

    return recommendations[:top_n]
```

- 残り3件は中受諾率だが、他の要因で離脱

#### 対処2: False Negative（見逃し）を減らす

**問題**: 高受諾率（> 30%）でも、最近の活動減少により見逃す

# 高受諾率の開発者には特別な配慮
if candidate['history_acceptance_rate'] > 0.30:

    if pred_prob < threshold:

**期待効果**:

# 中級者には厳しい基準

## 結論

### 主要な発見

1. **経験レベルが予測精度の最大の決定因子**
   - エキスパート（≥500回）: Accuracy 1.000（完璧）
   - 上級者（100-500回）: Accuracy 0.778（良好）
   - 中級者（10-100回）: Accuracy 0.615（困難）
   - 初心者（<10回）: 承諾者ゼロ

2. **「経験豊富 × 中受諾率」が最強の組み合わせ**
   - 上級者 × 中受諾率（10-25%）: 5名全員を完璧に予測
   - このセグメントは、最も推薦すべきグループ

3. **特徴量重要度の変化が予測精度に直結**
   - 経験の重要度が高い期間（0-3m）: 高精度（AUC-ROC 0.910）
   - 協力度の重要度が高い期間（9-12m）: 低精度（AUC-ROC 0.693-0.715）
   - 理由: 経験は安定した予測因子、協力度は変動しやすい

4. **予測失敗の5つのメカニズム**
   - 経験の過大評価
   - 中級者の不安定性
   - 最近の活動トレンドの見逃し
   - 協力度の時系列パターンの限界
   - 「復活パターン」の予測困難性

### 実務への推奨事項

**推薦すべき開発者**:
1. エキスパート（≥500回）全般
2. 上級者（100-500回）× 中受諾率（10-25%）
3. 中受諾率（10-25%）グループ全般

**推薦を避けるべき開発者**:
1. 初心者（<10回）
2. 未承諾（受諾率0%）
3. 超高受諾率（>40%）← 離脱の兆候
4. 中級者（10-100回）× 低受諾率（<10%）

**モデル選択**:
- 6-9ヶ月後の予測: 0-3m訓練モデル（AUC-ROC 0.910）
- 3-6ヶ月後の予測: 3-6m訓練モデル（AUC-ROC 0.820）
- リアルタイム推薦: 最新3-6ヶ月で訓練したモデル（月次更新）
